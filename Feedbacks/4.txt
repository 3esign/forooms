reating a persistent, web-based digital twin that dynamically translates OpenStreetMap (OSM) data into an interactive, participatory voxel environment is an ambitious and highly complex architectural challenge. To achieve this, the system must harmonize generative urbanism pipelines, multi-agent simulation, and stringent network constraints.To execute this vision successfully, we must immediately address the deployment architecture. While Vercel is exceptional for hosting the Next.js/Three.js frontend and stateless API routes, real-time multiplayer 3D games require persistent, long-lived WebSocket connections. Vercel’s serverless functions will drop these connections.Architectural Shift: Deploy the web client, UI, and standard APIs on Vercel. For the persistent multiplayer environment, utilize a managed real-time WebSockets infrastructure like PartyKit (built on edge workers, perfect for Next.js integrations) or host a dedicated Node.js Colyseus server on a containerized service like Railway or Fly.io. Both solutions maintain the open-source ethos while solving the persistence problem.Here is the comprehensive methodology for building the Foroom engine.1. Geospatial Data Integration & Transformation EngineThe process of converting vector OSM data into a 3D voxel grid requires precise geographic coordinate projection and procedural spatial rasterization.Fetching the Data: Utilize the Overpass API to query the specific bounding box selected on your 2D satellite map (Mapbox/MapLibre). Fetch nodes, ways, and relations filtered by building, highway, natural, and waterway tags.Coordinate Projection: Translate geospatial coordinates (Longitude $\lambda$, Latitude $\phi$) into a flat, local Cartesian grid ($x, z$) mapped to your 1m or 2m voxel scale.$$x = \lfloor R \cdot (\lambda - \lambda_0) \cdot \cos(\phi_0) \cdot \frac{\pi}{180} \cdot S \rfloor$$$$z = \lfloor R \cdot (\phi - \phi_0) \cdot \frac{\pi}{180} \cdot S \rfloor$$(Where $R$ is Earth's radius, $(\lambda_0, \phi_0)$ is the bounding box origin, and $S$ is the scale factor corresponding to your voxel resolution).Rasterization (The 2D Grid): Apply a point-in-polygon raycasting algorithm to fill the 2D bounding boxes of buildings and parks.Extrusion & Procedural Generation: Use OSM metadata (e.g., building:levels) to determine the height. Multiply the levels by 3 (assuming 3 meters per floor) and stack voxels along the $y$-axis. Apply procedural logic to map OSM tags to specific block textures (e.g., highway=residential to gray asphalt blocks, natural=tree to procedural voxel foliage).2. Voxel Data Optimization & State ArchitectureTo achieve your mandate of storing the "least data possible" while supporting live multiplayer, you must abandon standard relational database setups for block storage. Storing each block as a database row will cripple the application.Spatial Chunking: Divide the world into $32 \times 32 \times 32$ voxel chunks. The Three.js renderer will only instantiate chunks within the player's frustum (camera view).1D Array Serialization: Flatten the 3D grid of a chunk into a 1D TypedArray (Uint8Array if you have fewer than 255 block types). Calculate the index using:$$V_{index} = x + (y \times W) + (z \times W \times H)$$Run-Length Encoding (RLE): Most of the sky is empty ("air" blocks). Instead of storing 10,000 zeros, RLE compresses this into [0, 10000]. This reduces a chunk's memory footprint from 32KB down to mere bytes, which is then stored as a binary BLOB in your PostgreSQL database (e.g., Supabase).Delta Compression for Networking: When a Builder places a block, the server does not send the whole chunk to all players. It broadcasts a highly optimized binary payload over WebSockets containing only [x, y, z, blockID].3. The Layered Reality: Council vs. PlaygroundTo run two distinct digital twins without doubling your database storage, utilize a Base-and-Delta State Model.Council Mode (The Base): This is the immutable root generated from the OSM extraction. Only Builders and Admins can alter this geometry.Playground Mode (The Delta): When a Participant places or destroys a block in the Playground, do not duplicate the world. Instead, store a "Playground Diff" database record. When a user toggles to the Playground, the client takes the Council Base, applies the Playground Diff over it, and renders the result. This requires microscopic data storage.Glowing Heatmaps: For blocks containing notes or polls, inject metadata into a parallel key-value store where the key is the stringified coordinate ("x_y_z"). On the Three.js client, attach an .emissive material property to these specific blocks and utilize UnrealBloomPass for post-processing. Highly interacted blocks get a higher emissive intensity, creating the visual heatmap.4. This results in varying polygons (triangles, squares, pentagons, smooth cones) underneath the sphere. Multi-Agent AI Protector & Simulation OrchestrationIntegrating AI via OpenRouter allows the Foroom to become a living simulation, shifting from a static map to a generative spatial environment.Event-Driven Pulse: Do not allow the AI to run on a continuous requestAnimationFrame loop; this will instantly exhaust your API limits. Instead, hook the AI into an Event Queue (e.g., triggered every 10 block placements, or via a cron job every 5 minutes).Spatial Context Prompting: When querying the LLM, serialize a local radius of the voxel grid into a compressed text format. Feed the AI the current weather data, user chat logs, and block diffs.Automated Builder Actions: Give the AI "Builder" API credentials.
